## 2. Memory Architecture

**Why this design?**

The central insight is that research queries have *temporal structure*: earlier sub-questions provide background for later ones, but are not needed verbatim ‚Äî only their conclusions matter.  A three-layer hierarchy exploits this:

- **Layer 1** (working buffer) holds the hot context needed for the current LLM call ‚Äî compressed via summarisation before each call if the token estimate exceeds 75% of the per-call budget.
- **Layer 2** (episodic store) stores structured Episode objects, each with keyword metadata that enables relevance-scored retrieval.  This is much more efficient than a raw vector database for sessions with ‚â§ 10 episodes because it avoids embedding overhead.
- **Layer 3** (rolling summary) is a distilled narrative generated by a cheap LLM that replaces the episodic store once it grows beyond the `summarize_after` threshold.  This bounds memory growth to O(1) regardless of session length.

### 2.2 Alternative Considered: Vector RAG (ChromaDB / FAISS)

| Dimension | Hierarchical Episodic | Vector RAG |
|---|---|---|
| Retrieval quality | Keyword Jaccard ‚Äî fast, interpretable | Semantic embedding ‚Äî higher recall |
| Latency | Zero (in-memory dict lookup) | ~50-200 ms (embedding + ANN search) |
| Token overhead | None | Embedding model adds ~0.001¬¢/query |
| Persistence | Session-scoped (in-memory) | Persistent across sessions |
| Cold start | Instant | Needs index warm-up |
| Scalability | Degrades at >50 episodes | Scales to millions |

**Verdict for this use-case:** For a session-scoped research agent with ‚â§ 10 sub-queries, the episodic approach wins on latency and simplicity.  A vector database would be preferable for a long-running assistant that accumulates thousands of memories across sessions.

## 3. Token Constraint Strategy

### 3.1 Stated Constraints

| Constraint | Value | Enforcement Point |
|---|---|---|
| Max context tokens per call | 2,000 | `_maybe_compress()` in agent loop |
| Session token budget | 10,000 | `TokenBudget.is_exhausted()` |
| Summarisation trigger | 75% of per-call budget | `ResearchConfig.summarization_threshold` |
| Max sub-queries | 5 | `QueryDecomposer.decompose()` |
| Max iterations per sub-query | 3 | Inner agentic loop |

### 3.2 Compression Cascade

When the working buffer approaches the per-call limit, the agent runs a two-step cascade:

```
Step 1: Estimate token count (chars / 4 heuristic)
Step 2: If estimated > threshold:
          ‚Üí Send head (all but last message) to claude-haiku for compression
          ‚Üí Replace head with a single "compressed context" message
          ‚Üí Keep tail (last user message) intact
```

**Trade-off:** Compression introduces a lossy step.  The haiku model may drop edge-case facts.  This is acceptable because:
1. The synthesizer has access to all sub-answers directly, not just what survived compression.
2. The compression is only of the *conversational scaffolding*, not the episodic store.

### 3.3 Model Tier Selection

| Task | Model | Rationale |
|---|---|---|
| Query decomposition | claude-sonnet-4 | Needs good reasoning for structured output |
| Sub-query research | claude-sonnet-4 | Needs tool use capability |
| Summarisation / compression | claude-haiku-4-5 | Fast, cheap, sufficient for distillation |
| Final synthesis | claude-sonnet-4 | Quality matters for the user-facing answer |

Using haiku for compression saves ~60% of summarisation costs compared to using sonnet.

## 4. Query Decomposition

### 4.1 Design

The `QueryDecomposer` first applies a heuristic to detect simple queries (< 12 words, no conjunctions).  Simple queries skip decomposition entirely, saving one LLM round-trip.

For complex queries, the LLM is prompted to return a JSON array.  This is more reliable than free-text parsing and cheaper than few-shot examples because the output format constraint is part of the system instructions.

### 4.2 Trade-offs

| Approach | Pros | Cons |
|---|---|---|
| LLM-based JSON decomposition | Handles complex, nested queries | Adds 1 LLM call per research session |
| Rule-based (split on "and", "also") | Zero-cost | Brittle, misses semantic dependencies |
| Dependency graph (NLP parse) | Captures implicit structure | Requires NLP library, complex |

The LLM approach was chosen because the incremental cost (~100 input + 100 output tokens ‚âà < $0.001 per session) is negligible compared to the quality improvement it delivers on compound research questions.

## 5. Retrieval Quality vs. Cost

The retrieval mechanism uses **Jaccard similarity on keyword sets** extracted from episode queries and answers.  This is a deliberate choice:

**Why not embedding similarity?**

- Embedding a query and all episodes requires N+1 model calls (or a local embedding model).
- For a session with ‚â§ 10 episodes, keyword overlap is sufficiently discriminative.
- Keywords are extracted during episode storage (zero retrieval-time overhead).

**Known weaknesses of Jaccard retrieval:**
- Fails on synonyms (e.g., "car" vs "automobile").
- Ignores semantic context (e.g., "not safe" vs "safe").
- Sensitive to domain vocabulary (rare technical terms score well, common words score poorly).

**Mitigation:** Stopwords are filtered during tokenisation.  The retrieval budget means that even if retrieval is imperfect, the total tokens consumed remains bounded.

## 7. Failure Modes and Mitigations

| Failure | Impact | Mitigation |
|---|---|---|
| LLM decomposition returns invalid JSON | All queries collapse to 1 | Regex fallback + original query fallback |
| Session budget exhausted mid-session | Partial answer | Agent returns results for completed sub-queries |
| Tool raises exception | Missing evidence | Wrapped in try/catch; error string returned as tool result |
| Compression loses critical facts | Degraded synthesis quality | Synthesizer receives all sub-answers directly (not via memory) |
| haiku model unavailable | Compression fails | Falls back to string truncation |

## 8. Scalability Considerations

The current implementation is designed for **single-session, single-user** research.  For production multi-user scale:

1. **Replace in-memory job store** (`_jobs` dict in `app.py`) with Redis + Celery.
2. **Replace episodic store** with a persistent vector database (Pinecone, Weaviate) for cross-session memory.
3. **Add rate limiting** at the API layer (FastAPI middleware or NGINX).
4. **Instrument token budgets** per-user per-day with a quota system backed by a SQL database.
5. **Cache decomposition results** for identical or near-identical queries (hash the query, store in Redis with TTL).

## 10. Summary of Key Decisions

1. **Hierarchical episodic memory** over a flat vector database ‚Äî best fit for session-scoped, token-constrained research.
2. **LLM-based JSON decomposition** over rule-based parsing ‚Äî higher quality at negligible extra cost.
3. **Jaccard keyword retrieval** over embedding search ‚Äî zero latency, sufficient quality for ‚â§ 10 episodes.
4. **Dual-model strategy** (sonnet for reasoning, haiku for compression) ‚Äî optimises cost without sacrificing output quality.
5. **Compression cascade** triggered at 75% budget ‚Äî prevents context overflow while preserving the most recent (most relevant) messages.
6. **Graceful degradation** at every failure point ‚Äî the agent never raises uncaught exceptions to the user.
